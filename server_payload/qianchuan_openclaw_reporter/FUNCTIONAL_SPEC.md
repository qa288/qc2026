## 2. 当前系统目标

系统目标不是做一个通用 BI，而是做一个面向老板和运营主管的千川实时决策面板。

第一版聚焦工作台账号：

- `1802375893828620`

需要解决的问题：

- 实时查看整体经营情况
- 实时查看每个账户的投流表现
- 实时查看每个计划和商品的投流效果
- 实时查看员工和商品两个经营维度的排名
- 基于规则触发提醒
- 按配置时间推送简报
- 保留分钟级历史快照供复盘

## 3. 用户角色

### 3.1 老板

老板主要关心：

- 整体消耗
- 整体支付
- 整体订单
- 整体 ROI
- 当前是否有异常提醒
- 哪些账户或计划需要立即关注

### 3.2 运营主管

运营主管主要关心：

- 账户之间的差异
- 计划之间的排序和比较
- 具体商品计划效果
- 当前投放状态
- 提醒规则是否合理
- 是否需要暂停、放量、调价、换品

### 3.3 系统维护者

系统维护者主要关心：

- 官方 API 是否正常
- token 刷新是否正常
- 页面是否可访问
- 通知渠道是否配置正确
- 定时链路是否按时执行

## 4. 当前技术拓扑

### 4.1 数据源

只允许使用巨量官方 `open_api`。

当前核心数据来源：

- `oauth2/refresh_token`
- `customer_center/advertiser/list`
- `qianchuan/report/uni_promotion/get`
- `qianchuan/uni_promotion/list`

当前已进入正式落库但尚未全部进入正式页面展示的数据来源：

- `qianchuan/uni_promotion/ad/detail`
- `qianchuan/uni_promotion/ad/product/get`
- `qianchuan/uni_promotion/ad/material/get`
- `qianchuan/file/video/original/get`

当前已用于能力发现、未默认进入周期性通用主题落库的数据来源：

- `qianchuan/report/uni_promotion/config/get`
- `qianchuan/report/uni_promotion/data/get`

### 4.2 页面服务

- 技术栈：`FastAPI + Jinja2 + SQLite + 原生 JS/CSS`
- 部署方式：`Docker Compose`
- 容器名：`qianchuan-dashboard`
- 端口：`9898`
- 访问方式：公网访问 + 登录保护

### 4.3 通知服务

- 脚本：`bridge_send_alerts.py`
- 发送方式：`openclaw message send`
- 发送目标：页面配置决定
- 定时器：`qianchuan-dashboard-alerts.timer`
- 当前检查频率：每 `1` 分钟

### 4.4 数据存储

- 数据库：`/opt/qianchuan-dashboard/data/dashboard.db`
- 存储内容：
  - 总体快照
  - 账户快照
  - 计划快照
  - 计划详情快照
  - 计划商品快照
  - 计划素材快照
  - 视频首发标记
  - 细粒度同步状态
  - 告警规则
  - 告警事件
  - 通知设置
  - 通知发送日志

## 5. 不可回退的系统约束

以下要求为重构硬约束，除非用户明确要求，否则不能改掉：

- 只能使用官方 `open_api`
- 不允许改回浏览器抓取、Cookie、`/nbs/api/*`
- 必须保留桌面优先网页
- 必须保留 `9898` 端口和登录保护
- 必须保留分钟级历史快照
- 必须保留细粒度扩展快照，至少包括计划详情、商品、素材、视频首发标记
- 必须保留账户层和计划层明细
- 必须保留商品层和员工层排名
- 必须保留提醒规则
- 必须保留可配置通知渠道、通知目标、定时推送时间
- 必须保留自动排序刷新后不重置当前排序
- 必须保留计划状态中文显示
- 员工归属第一版必须保持“按计划里的主播 / 抖音号字段归属，空值归未归属”的默认口径，除非后续显式引入映射表
- 新增任何数据前必须先经过能力发现，不允许直接跳过目录和发现脚本猜接口

## 6. 功能需求与当前实现

本节按模块列出“必须具备的功能”与“当前已实现的行为”。

### 6.1 登录页

需求：

- 必须有登录保护
- 登录页必须是桌面端友好的视觉，不是默认表单
- 输入框和按钮尺寸要足够大，适合日常使用

当前实现：

- 登录地址：`/login`
- 账号密码校验来自环境变量
- 登录页是左右双区布局
- 输入框高度：大尺寸输入框
- 登录成功后进入首页

### 6.2 首页顶区

需求：

- 顶区需要保持简洁
- 不要堆过多介绍文案
- 要突出这是实时经营看板

当前实现：

- 顶区只有标题、简短说明和两个操作按钮
- 按钮包括：
  - `立即刷新`
  - `退出登录`

### 6.3 KPI 总览

需求：

- 首页必须在第一屏给出核心经营数字

当前实现：

- 今日消耗
- 今日 ROI
- 活跃账户 / 总账户
- 活跃计划 / 总计划
- 活跃商品 / 总商品
- 活跃员工 / 总员工
- 账户查询异常数
- 计划查询异常数

### 6.4 老板关注模块

需求：

- 不能出现明显右侧留空
- 更适合横屏
- 同时展示核心提醒摘要和最近命中记录

当前实现：

- 模块已改成横屏两区布局
- 左侧：
  - 最近触发
  - 待处理数量
  - 最近对象
- 右侧：
  - 最近提醒列表
  - 横向多卡片自适应排列

### 6.5 最新提醒列表

需求：

- 不能只显示一段原始文字
- 必须清楚显示对象、规则、当前值和状态

当前实现：

- 每条提醒卡片显示：
  - 对象类型
  - 指标
  - 处理状态
  - 触发时间
  - 规则条件
  - 当前值
  - 消耗
  - 支付
  - 订单
  - ROI
  - 原始提醒文案

### 6.6 账户表现表

需求：

- 必须能按账户查看基本情况
- 必须支持搜索
- 必须支持点击排序
- 必须支持按 `日 / 周 / 月` 切换真实查询范围

当前实现：

- 字段：
  - 账户名
  - 消耗
  - 支付
  - 订单
  - ROI
  - 状态
- 支持账户名搜索
- 支持排序
- 支持 `日 / 周 / 月`
- 当前不再展示账户趋势快照或账户详情面板，账户区保持纯表格，避免干扰老板看板主路径

### 6.7 商品与员工排名

- 必须新增员工维度实时排名
- 必须新增商品维度实时排名
- 两个维度都必须支持搜索
- 两个维度都必须支持点击排序
- 两个维度都必须支持 `日 / 周 / 月`
- 刷新后必须保持当前排序
- 点击行后必须能看该员工 / 商品的聚合详情

  - 支付
  - 订单
  - ROI
  - 消耗
  - 账户数
  - 商品数
  - 计划数
- 商品排名字段：
  - 商品
  - 订单
  - 支付
  - ROI
  - 消耗
  - 账户数
  - 员工数
  - 计划数
- 员工详情显示：
  - 消耗

  - 订单
  - ROI

  - 覆盖商品
  - 总计划数

### 6.8 计划效果明细表

需求：

- 计划层必须是运营决策主表
- 必须支持按计划/商品/账户/主播搜索
- 必须支持点击排序
- 刷新后必须保持当前排序方式
- 必须支持点击计划看详情

当前实现：

- 字段：
  - 计划
  - 商品
  - 营销目标
  - 账户
  - 主播
  - 订单
  - ROI
  - 目标 ROI
  - 消耗
  - 支付
  - 投放状态
- 支持搜索
- 支持账户筛选
- 支持 `日 / 周 / 月`
- 支持本地排序记忆
- 点击计划可查看计划详情

### 6.9 计划详情

需求：

- 必须能看具体商品计划的投流效果
- 必须展示计划核心业务信息和状态

当前实现：

- 字段：
  - 计划名
  - 商品名
  - 所属账户
  - 主播
  - 统计订单
  - 统计 ROI / 目标
  - 统计消耗
  - 统计支付
  - 投放状态
  - 营销目标
- 不再展示趋势图，详情只保留对投流决策有帮助的字段面板

### 6.10 投放状态中文化

需求：

- 投放状态必须显示中文
- 不能直接显示接口枚举值
- 必须保证状态来源正常，不是前端瞎猜

当前实现：

- 后端基于官方接口真实状态值映射中文状态
- 当前线上已验证状态枚举：
  - `DELIVERY_OK`
  - `DISABLE`
  - `SYSTEM_DISABLE`
  - `ENABLE`
  - `ROI2_DISABLE`
- 当前中文映射：
  - `DELIVERY_OK + ENABLE` -> `投放中`
  - `DISABLE + DISABLE` -> `已暂停`
  - `SYSTEM_DISABLE + ROI2_DISABLE` -> `系统暂停`
- API 额外返回：
  - `status_label`
  - `opt_status_label`
  - `status_text`
  - `status_code_text`

### 6.11 通知与告警设置

需求：

- 页面里必须能配置是否推送
- 通知渠道必须可选
- 通知目标必须可改
- 必须能设置定时推送时间
- 必须能配置简报里展示多少账户和多少计划

当前实现：

- 配置项：
  - 开启通知推送
  - 通知渠道
  - 渠道账号
  - 通知目标
  - 推送阈值告警
  - 告警批量条数
  - 推送定时数据简报
  - 推送时间
  - 简报账户条数
  - 简报计划条数
- 页面右侧显示当前通知状态摘要

### 6.12 告警规则

需求：

- 告警规则必须可在页面中直接维护
- 不能写死在代码里

当前实现：

- 规则字段：
  - 对象
  - 指标
  - 比较符
  - 阈值
  - 最低消耗门槛
  - 冷却分钟
  - 限定 ID
  - 备注
  - 启用/关闭
- 支持新增、启停、删除

### 6.13 告警事件

需求：

- 页面提醒不能只显示“当前状态”
- 必须保留历史命中记录

当前实现：

- 每条命中写入 `alert_events`
- 页面读取最近事件倒序展示
- 发送后事件状态会标成 `sent`

### 6.14 定时通知

需求：

- 支持阈值提醒
- 支持定时简报
- 定时简报时间必须由页面配置

当前实现：

- `bridge_send_alerts.py` 每分钟执行一次
- 通知关闭时不会外发
- 通知开启后：
  - 告警按待发送事件批量推送
  - 简报按 `HH:MM` 配置时间触发
- 为避免一开开关就补发很久以前的积压提醒，当前只外发最近一段时间的新待发送告警

### 6.15 历史快照

需求：

- 必须保留分钟级历史数据
- 必须能支撑趋势回看

当前实现：

- 页面服务每 `1` 分钟采集一次
- 历史保留 `180` 天
- 账户和计划详情都使用历史快照绘图

- 消耗

```text
总支付金额 / 总消耗
```


- `stat_cost`
- `total_pay_order_gmv_for_roi2`

## 14. 重构后必须通过的验收清单

每次做较大修改后，至少要验证以下项目：

### 14.1 页面访问

- `http://服务器IP:9898/login` 可访问
- 正确账号密码可登录
- 首页能正常加载

### 14.2 数据显示

- KPI 卡片有值
- 账户表有值
- 计划表有值
- 点击账户有详情
- 点击计划有详情

### 14.3 状态显示

- 计划状态显示中文
- 至少能看到 `投放中 / 已暂停 / 系统暂停`
- 不允许直接展示 `DELIVERY_OK / DISABLE / SYSTEM_DISABLE`

### 14.4 提醒区

- 老板关注模块不出现明显右侧空白
- 摘要区和提醒区都正常展示

### 14.5 通知设置

- 页面能看到通知设置表单
- 能保存渠道、目标、推送时间
- 保存后数据库有对应值

### 14.6 历史数据

- 新快照持续写入
- 历史详情接口可返回数据

### 14.7 通知链路

- timer 处于激活状态
- 通知关闭时不会外发
- 通知开启且命中条件时可以外发
