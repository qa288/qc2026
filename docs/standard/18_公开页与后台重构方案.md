# 18 公开页与后台重构方案

最后更新：`2026-03-22`

本方案用于替换旧的“单登录主看板”范围。  
后续 UI、路由、权限、归属和预警开发，以本文档为当前执行方案。

---

## 1. 目标

本轮重构的目标不是继续在现有页面上补零碎功能，而是统一成一套清晰的产品结构：

1. 打开网站即可看到公开员工排名页。
2. 登录后进入内部工作台，处理账户、计划、素材、归属、权限和预警。
3. 员工归属从当前临时的 `anchor_name` 聚合，升级为“关键词 + 人工修正”的正式模型。
4. 预警中心第一阶段完成：
   - 规则配置
   - 预警引擎
   - 记录与状态
   - 多通知渠道配置结构
   - 账户余额 / 共享钱包 / 消耗 / 爆单规则
   - 实际通知发送可后置开启
5. 页面风格延续当前视觉方向，但重新规划布局、信息层级和操作顺序。

---

## 2. 路由与角色

### 2.1 路由

- `/`
  - 公开员工榜
- `/login`
  - 后台登录
- `/workbench`
  - 内部工作台入口
- `/api/*`
  - 内部接口

### 2.2 角色

- 匿名访问者
  - 只能看公开员工榜
- 运营
  - 与管理员使用同一工作台
  - 只能看授权账户范围
- 管理员
  - 看全量数据
  - 管理员工、规则、权限、系统配置

### 2.3 角色原则

- 匿名页只给脱敏后的员工榜，不暴露账户余额、钱包信息、系统状态、Token、规则配置。
- 运营和管理员登录后看到的是同一套工作台，但数据范围不同。
- 计划、素材、商品权限全部继承账户权限。

---

## 3. 公开页方案

### 3.1 页面目标

公开页只做一件事：  
**让人打开网址就能看到员工排名。**

### 3.2 第一阶段内容

- 员工名称
- 排名序号
- 总消耗
- 支付金额
- 订单量
- ROI
- 计划数
- 账户数
- 可切换排序方向
- 可切换时间：
  - 今日
  - 近 7 天
  - 当月
  - 自定义时间段
- 每分钟自动刷新

### 3.3 第一阶段不展示

- 计划明细
- 素材明细
- 账户余额
- 共享钱包
- 告警记录
- 内部配置入口

### 3.4 交互原则

- 首屏直接是榜单，不要说明文案占大量高度。
- 默认按消耗降序。
- 搜索默认隐藏，可放在工具栏折叠入口或次级区域。
- 排序切换不丢失当前时间范围。

## 4. 后台工作台方案

## 4.1 页面结构

内部工作台统一为 7 个模块：

1. 总览
2. 账户
3. 计划
4. 素材
5. 员工归属
6. 预警中心
7. 权限与授权

### 4.2 总览

用于老板和运营主管快速判断：

- 今天整体消耗 / 支付 / 订单 / ROI
- 活跃账户 / 计划 / 商品 / 员工
- 重点异常
- 最近一次同步状态

要求：

- 去掉重复标题
- 不再出现大面积空白
- 状态信息收进右侧细栏或顶部一行
- 指标主次分明，核心数据一屏看完

### 4.3 账户页

- 账户表格
- 时间筛选
- 排序
- 搜索
- 账户详情抽屉

详情里只放“补充信息”，不要重复左表已有字段。

### 4.4 计划页

- 计划主表
- 顶部时间筛选与账户过滤
- 右侧详情抽屉
- 右侧商品 / 素材摘要

优化原则：

- 日期区先选时间，再筛账户，再搜索
- 右侧说明性字段减少
- 重复字段从详情移除

### 4.5 素材页

- 素材表格
- 搜索
- 右侧素材表现详情

优化原则：

- 左侧直接展示 `素材名 + 素材 ID + 视频 ID`
- ID 中间截断，悬停显示完整值
- 右侧不重复展示名称和 ID，只展示关系和表现

### 4.6 员工归属页

这是本轮新增重点页面。

页面拆成三列：

1. 员工列表
2. 关键词与命中结果
3. 人工修正区与未归属池

支持：

- 新增员工
- 停用员工
- 给员工配置多个关键词
- 查看关键词命中的账户 / 计划 / 素材
- 勾选命中结果加入员工归属
- 查看未归属对象池
- 直接从未归属池绑定账户 / 计划 / 商品

### 4.7 预警中心

第一阶段先把结构做完整，通知发送可按开关关闭。

分成三块：

1. 快速模板 + 新增 / 编辑规则
2. 通知渠道配置
3. 规则列表

通知页要改成：

- 先选模板，再补阈值
- 渠道配置放次级区
- 规则列表放底部

不要把“通知设置”和“规则设置”互相打散。

### 4.8 权限与授权页

支持：

- 运营账号列表
- 为运营账号绑定账户范围
- 查看某个运营实际可见范围

第一阶段不做复杂 RBAC，只做够用的账户授权模型。

## 5. 员工归属模型

## 5.1 当前问题

当前系统只是按计划里的 `anchor_name` 临时聚合员工。  
这个只能做演示，不能作为正式统计口径。

## 5.2 正式模型

员工归属采用两层：

1. 关键词命中
2. 人工修正

### 5.3 数据对象

关键词可以匹配四类对象：

- 账户名
- 计划名
- 商品名
- 素材名

### 5.4 规则

- 同一个员工可配置多个关键词
- 搜索范围支持：
  - 全部
  - 账户
  - 计划
  - 商品
  - 素材
- 同一个关键词命中的多条账户/计划/商品/素材，管理员可以勾选纳入员工归属
- 搜索结果允许多员工命中
- 排名统计只允许单归属

### 5.5 归属优先级

正式优先级定为：

1. 人工素材归属
2. 人工计划归属
3. 人工账户归属
4. 关键词命中素材
5. 关键词命中计划
6. 关键词命中账户
7. `anchor_name` 临时兜底
8. 未归属

### 5.6 页面表现

- 搜索结果页看到命中对象后，可以直接勾选归属给员工
- 员工榜统计时只按最终归属表计算
- 未归属对象必须有单独列表

## 6. 预警范围与通知策略

## 6.1 第一阶段要做的预警类型

尽管手机通知可以暂时不开，但这些规则和引擎要先做出来：

- 通用余额预警
- 共享钱包余额预警
- 消耗阈值预警
- 爆单预警

### 6.2 爆单预警口径

默认按：

- 单计划
- 指定时间窗口内
- 支付订单量超过阈值

### 6.3 第一阶段通知策略

- 规则配置和记录先做完整
- 多通知渠道配置结构先做完整
- 实际发送通道可先关闭
- 后续再按开关启用：
  - Feishu
  - DingTalk
  - Email
  - Slack
  - Telegram

### 6.4 页面原则

- 通知配置与规则编辑合并在一个模块内
- 操作顺序应是：
  1. 选通知渠道
  2. 配默认目标
  3. 新增规则
  4. 查看记录
- 不要把“通知”和“规则”做成互相孤立的两块

## 7. 后端架构落地

仍采用已确认的生产架构：

- Web/API：FastAPI
- 主数据库：PostgreSQL
- 缓存 / 分布式锁 / 队列：Redis
- 异步任务：Celery
- 调度：独立 scheduler / Celery Beat
- 部署：Docker Compose

### 7.1 新增后端模块

需要新增或扩展：

- `public rankings`
  - 公开员工榜接口
- `authz`
  - 运营账号与账户授权
- `employee mapping`
  - 员工、关键词、人工归属
- `alert engine`
  - 余额 / 钱包 / 消耗 / 爆单
- `notification adapters`
  - 通知渠道配置和适配层

### 7.2 数据存储新增

在现有表基础上新增：

- `employees`
- `employee_keywords`
- `employee_manual_bindings`
- `operator_accounts`
- `operator_account_scopes`
- `account_balances`
- `shared_wallets`
- `shared_wallet_account_relations`
- `alert_rule_runs`

### 7.3 查询策略

- 公开页优先查聚合表 / 缓存
- 内部榜单优先查本地快照和聚合结果
- 页面不直接打官方接口
- 余额 / 钱包同步走独立任务

---

## 8. UI 重构原则

## 8.1 保留

- 当前蓝白工作台风格
- 当前轻前端方向
- 当前表格、侧栏、搜索交互的基本逻辑

## 8.2 必须重构

- 标题层级
- 模块布局
- 操作顺序
- 大块留白
- 重复字段
- 表单区结构

## 8.3 统一规则

1. 一个模块只保留一个主标题。
2. 左表已有字段，右侧详情不重复。
3. 主操作从左到右、从上到下，不能让用户来回跳视线。
4. 状态、说明、帮助文案尽量短。
5. 详情抽屉只承载补充信息和动作，不承载重复信息。

## 10. 当前默认执行口径

如果没有额外变更，后续开发按以下默认值执行：

- 做公开员工榜
- 不做大屏
- 不做定时通知
- 做权限模型
- 做员工归属完整第一版
- 做余额 / 共享钱包 / 爆单预警
- 审计日志后置
- 通知通道配置先做，实际通知可后置开启

## 11. 仍建议确认的细节

当前还有 4 个细节建议在进入开发前最终确认：

1. 公开页默认显示：
   - 排名
   - 归属人名称
   - 总消耗
   - 支付金额
   - 订单量
   - ROI
2. 爆单预警第一阶段默认按计划。
3. 运营账号体系第一阶段默认使用本地账号密码。
4. 归属人展示统一使用管理员配置的展示名。
