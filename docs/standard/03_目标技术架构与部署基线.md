# 03 目标技术架构与部署基线

## 1. 目标

本文档定义 V1 正式开发的唯一目标架构。  
后续开发、部署、运维如果与本文档冲突，以本文档为准。

---

## 2. 统一技术栈

第一阶段正式技术栈固定为：

- Web/API：`FastAPI`
- 主数据库：`PostgreSQL`
- 缓存 / 队列 / 分布式锁：`Redis`
- 异步任务：`Celery`
- 定时调度：独立 `scheduler` 进程，可采用 `Celery Beat`
- 部署：`Docker Compose`
- 前端：保留当前轻前端，不强制前后端分离
- 反向代理与运维入口：`1Panel`

以下方案降为历史方案，不再作为正式要求：

- `Next.js`
- `NestJS`
- `Prisma`
- `BullMQ`
- `TypeScript Monorepo`

---

## 3. 服务边界

运行角色统一如下：

### 3.1 web

职责：

- 登录与会话
- 页面渲染
- 内部 API
- 只读查询

禁止：

- 在 Web 进程里承担正式业务调度

### 3.2 worker

职责：

- 官方接口采集
- 数据落库
- 聚合计算
- 告警评估与通知任务执行

### 3.3 scheduler

职责：

- 发起定时任务
- 不承担页面请求

### 3.4 postgres

职责：

- 结构化数据真源

### 3.5 redis

职责：

- Celery Broker
- 缓存
- 分布式锁
- Token 刷新锁

---

## 4. 数据与存储分层

### 4.1 PostgreSQL

保存：

- 账户、计划、商品、素材结构化数据
- 快照与聚合结果
- 用户、规则、告警、通知、任务日志
- Token 元数据

### 4.2 本地卷 / 对象存储

保存：

- 原始接口回包
- 导出文件
- 调试快照

第一阶段允许先使用本地卷；后续可切对象存储。

### 4.3 Token 缓存文件

规则：

- PostgreSQL 为主真源
- 本地文件为缓存副本
- 推荐路径：`/app/data/qianchuan_latest_token.json`

---

## 5. Token 标准

### 5.1 刷新规则

- 只允许一个实例刷新 token
- 刷新成功后必须立即保存新的 `refresh_token`
- 其它实例只能读取最新 token

### 5.2 共享方式

标准方式：

- `GET /api/system/integrations/ocean-engine/token-latest`

补充方式：

- 只读缓存文件

禁止：

- 多台服务器各自刷新同一套 `refresh_token`

---

## 6. 账户汇总降级标准

标准链路：

1. 优先调用官方账户汇总接口
2. 若返回 `50000` 或持续不可用
3. 自动降级为计划聚合账户汇总

要求：

- 明确标记来源
- 不得把降级值伪装成官方原始值

---

## 7. 通知基线

第一阶段只保留：

- 阈值告警

通知通道：

- 系统 -> 通知适配层 -> OpenClaw / 其它渠道

后续扩展：

- Feishu
- DingTalk
- Email / Slack / Telegram

第一阶段不做：

- 定时简报
- 日报 / 周报
- 多通道联动

---

## 8. 资源基线

### 8.1 试运行 / 联调

- `2C4G`

### 8.2 正式生产（目标 1000 账户）

建议至少：

- `8C16G`
- SSD 持久化卷
- PostgreSQL、Redis、应用容器独立资源控制

说明：

- `2C4G` 只适合作为试运行起步配置
- 不应作为 `1000` 账户正式容量基线

---

## 9. 当前实现与目标架构的关系

现网试运行实现主要用于：

- 验证官方接口能力
- 验证页面流程
- 验证同步和通知链路

正式开发时：

- 试运行实现可以复用经验
- 但目标生产架构仍以本文档为准

---

## 10. 现网运维补充

当前服务器已安装 `1Panel`，用于：

- 站点反向代理
- HTTPS 证书管理
- Docker / Compose 应用管理
- 基础运维查看

约束：

- `1Panel` 是现网运维入口，不替代应用内部架构
- 应用运行形态仍然是 `web / worker / scheduler / postgres / redis`
- 文档中提到的 `Docker Compose` 仍是应用编排基线
