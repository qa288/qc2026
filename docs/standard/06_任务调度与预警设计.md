# 06 任务调度与预警设计

## 1. 目标

调度系统负责稳定同步官方数据，并驱动：

- 数据采集
- 聚合计算
- 榜单刷新
- 阈值告警
- 通知发送

第一阶段采用轮询同步，不接浏览器抓取，不做定时简报。

---

## 2. 调度原则

1. Web 不承担正式业务调度。
2. scheduler 只发任务，worker 负责执行。
3. 高频任务与低频任务分离。
4. 每类任务必须可重试、可追踪、可审计。
5. 预警计算基于本地数据，不直接实时打官方接口。

---

## 3. 调度框架

统一采用：

- 定时调度：独立 `scheduler` 进程
- 异步执行：`Celery`
- Broker / 锁：`Redis`

不再使用：

- `NestJS Scheduler`
- `BullMQ`

---

## 4. 核心任务清单

### 4.1 `token.refresh`

- 作用：刷新和维护最新 token
- 规则：只允许一个实例执行

### 4.2 `sync.accounts`

- 作用：同步工作台下账户结构
- 频率：低频

### 4.3 `sync.plans`

- 作用：同步计划结构和计划快照
- 频率：分钟级或近分钟级

### 4.4 `sync.materials`

- 作用：同步素材与视频明细
- 频率：低于计划同步频率

### 4.5 `sync.reports`

- 作用：同步账户、计划、商品、素材相关报表
- 频率：分钟级

### 4.6 `rebuild.rankings`

- 作用：刷新榜单缓存和聚合结果

### 4.7 `alerts.evaluate`

- 作用：评估阈值规则
- 频率：分钟级

### 4.8 `alerts.dispatch`

- 作用：发送阈值告警通知

### 4.9 `cleanup.*`

- 作用：清理过期快照、导出文件、临时数据

---

## 5. Token 任务规则

1. 只允许单点刷新
2. PostgreSQL 保存 token 真源
3. 本地 token 文件只作为缓存副本
4. Redis 锁防止并发刷新
5. 其它实例只读最新 token

---

## 6. 账户汇总降级

### 标准链路

1. 先调官方账户汇总
2. 若 `50000` 或持续失败
3. 自动回退为计划聚合

### 要求

- 失败不阻断整批同步
- 要保留 request_id
- 要标记聚合来源

---

## 7. 告警范围

第一阶段只支持 **阈值告警**：

- 账户阈值告警
- 计划阈值告警
- 素材阈值告警

第一阶段不做：

- 定时简报
- 日报
- 周报
- 多渠道分级通知

---

## 8. 通知规则

当前链路：

- 系统 -> OpenClaw -> Feishu

后续扩展：

- 仅替换 adapter 配置切换到 DingTalk

---

## 9. 调度完成标准

- 同步任务失败不影响 Web 可用性
- 单账户失败不影响全局
- 单计划失败不影响其它计划
- 告警评估与通知可追踪
- Token 刷新冲突可控
