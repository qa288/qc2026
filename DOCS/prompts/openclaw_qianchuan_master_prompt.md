# OpenClaw 主提示词：巨量千川官方采集、定时推送与增量扩展

> 说明：这是一份历史初始化模板，主要对应早期 `SQLite + OpenClaw cron + 日报/简报` 方案。  
> 当前正式开发与部署基线已经改为 `FastAPI + PostgreSQL + Redis + Celery + 独立 scheduler + Docker Compose`。  
> 后续如需继续使用提示词，应优先参考 [目标技术架构与部署基线](../standard/03_目标技术架构与部署基线.md) 和 [已确认决策基线](../standard/13_已确认决策基线.md)，不要直接按本文原样落地。

---

## 1. 主提示词

```text
你要在本机实现一个“巨量千川数据采集、定时推送、阈值告警”系统。严格按以下要求执行，不要自行改架构，不要擅自换成浏览器抓取方案。

一、总体目标
1. 定时从巨量千川官方开放平台拉取工作台账号下全部子账户的数据。
2. 将原始数据和汇总数据稳定保存到本地。
3. 定时把我指定的数据摘要发送到已经配置好的 OpenClaw 内置 Feishu。
4. 按预设规则做数据触发提醒，并发送到 Feishu。
5. 所有任务必须可重复执行、可恢复、可追踪，不依赖聊天上下文记忆。

二、强约束
1. 只使用巨量官方 open_api，不使用 business.oceanengine.com 页面接口，不使用浏览器自动化，不使用 Cookie、csrftoken、/nbs/api/bm/*。
2. 千川数据接口只使用这组：
   - oauth2/access_token
   - oauth2/advertiser/get
   - customer_center/advertiser/list
   - qianchuan/report/uni_promotion/get
   - qianchuan/uni_promotion/list
   - qianchuan/uni_promotion/ad/detail
   - qianchuan/report/uni_promotion/config/get
   - qianchuan/report/uni_promotion/data/get
3. 本地存储使用 SQLite。
4. 代码使用 Python 3.11。
5. 定时任务使用 OpenClaw Gateway 内置 cron，不使用系统 crontab。
6. 所有 cron 任务使用 isolated session。
7. 所有脚本必须幂等；重复运行不能产生脏数据。
8. 任何 ROI、成本、退款率类字段不得直接跨账户平均，必须由分子分母重算。
9. 所有敏感信息必须放到配置文件或环境变量，不允许硬编码。
10. 所有错误必须写入本地日志。

三、目录结构
请按以下结构生成：
project/
  config/
    config.example.json
  data/
    qianchuan.db
    raw/
  logs/
  scripts/
    sync_accounts.py
    sync_account_summary.py
    sync_plan_list.py
    sync_plan_detail.py
    build_daily_report.py
    check_alerts.py
    send_feishu_report.py
    run_full_sync.py
  DOCS/
    README.md

四、本地数据表
至少创建这些表：
1. customer_center_accounts
2. advertiser_accounts
3. uni_account_summary
4. uni_plan_summary
5. uni_plan_detail_latest
6. alert_rules
7. alert_events
8. job_runs

五、采集逻辑
1. 先从 customer_center/advertiser/list 获取工作台账号下全部子账户。
2. 对每个子账户拉账户维度汇总。
3. 对每个子账户拉计划列表。
4. 对新增或更新的计划拉计划详情。
5. 原始接口返回同时保存到 data/raw。
6. 汇总结果写入 SQLite。
7. 工作台总表由子账户数据自行汇总生成。

六、时间策略
1. 每小时同步最近 7 天账户汇总。
2. 每小时同步最近 7 天计划列表。
3. 每天凌晨重刷最近 30 天。
4. 每周重刷最近 180 天。
5. 告警任务每 10 分钟运行一次。
6. 每天早上 09:00 生成并推送日报。
7. 时区固定为 Asia/Shanghai。

七、告警规则
先实现这 6 类：
1. 单账户今日消耗超过阈值。
2. 单账户今日 ROI 低于阈值。
3. 单计划今日消耗超过阈值。
4. 单计划 ROI 低于阈值。
5. 有消耗但 2 小时内无订单。
6. 接口连续失败达到阈值。

规则要求：
1. 阈值放在 alert_rules 表中配置。
2. 同一对象同一规则在冷却时间内只提醒一次。
3. 告警消息必须包含：时间、账户、计划、指标、阈值、当前值、建议动作。

八、Feishu 推送要求
1. 使用已配置好的 OpenClaw Feishu channel。
2. 默认推送到一个固定目标，由配置文件指定。
3. 支持群 chat_id 或用户 open_id。
4. 日报和告警都输出 Markdown 风格纯文本，避免复杂卡片。
5. 单条消息超长时自动分段。

九、OpenClaw cron 任务
请为我设计并创建这些 cron 任务：
1. qianchuan-sync-hourly
2. qianchuan-alert-check
3. qianchuan-daily-report
4. qianchuan-backfill-nightly

要求：
1. 所有 cron 任务都记录 run 日志。
2. 同一时刻不允许并发执行多个采集任务。
3. 失败时保留错误信息。
4. 如果 OpenClaw 当前版本存在旧 cron 格式兼容问题，请提醒我执行 doctor 修复。

十、增量扩展规则
以后如果我说“新增某些数据、报表、组合指标、提醒逻辑”，你必须在现有系统上增量扩展，不允许重写整个系统。

你必须严格遵守以下规则：
1. 不要重写现有官方 open_api 主链路。
2. 不要改成浏览器抓取、页面接口、Cookie 方案。
3. 优先复用现有表、脚本、任务；只有确实不够时才新增表或脚本。
4. 任何新增字段、维度、指标，都必须先检查：
   - 属于哪个数据对象：工作台、子账户、计划、商品、素材、标题、图文、视频、直播间
   - 用哪个官方接口实现
   - 是否需要先调用 uni_promotion/config/get 确认可用字段
5. 如果新增需求需要改表结构，必须先给出：
   - 表结构变更方案
   - 数据迁移方案
   - 回填方案
6. 不允许破坏现有 cron 任务和历史数据结构兼容性。
7. 所有新增逻辑必须保持幂等。
8. 所有新增数据仍然要保留 raw_json。
9. 所有新增提醒规则必须支持去重和冷却时间。
10. 如果某个接口字段组合不稳定，请拆批查询，不要因为一次 50000 就中断整条任务链路。

十一、以后我追加需求时，你的标准处理流程
当我说“给我新增某某数据”时，你必须按这个顺序执行：
1. 先识别需求的：
   - 数据对象
   - 时间范围
   - 粒度
   - 维度
   - 指标
   - 输出位置
   - 是否需要告警
2. 再判断使用哪个官方接口。
3. 再判断是否需要：
   - 新字段
   - 新表
   - 新脚本
   - 新 cron
4. 先给我一个最小变更方案。
5. 再实施代码变更。
6. 最后给我：
   - 变更点
   - 新增表或字段
   - 回填方式
   - 如何验证

十二、以后我说“新增数据”时，如果我没有说全，你必须按以下结构补齐并执行
默认理解模板：
1. 数据对象：账户、计划、商品、素材中最接近的一类
2. 时间范围：最近 7 天
3. 粒度：按天
4. 输出：落库 + 可加入日报，不默认加告警
5. 推送：不主动推送，除非我明确要求

如果我的描述过于模糊，你先给出“你将采用的默认解释”，然后按最小可运行版本实施。

十三、交付方式
每次新增功能时，按以下顺序输出：
1. 需求理解
2. 接口选择
3. 最小变更方案
4. 表结构变更
5. 代码变更
6. cron 变更
7. 验证方法

十四、额外要求
1. 先做最小可运行版本，不要一开始追求复杂 UI。
2. 任何不确定字段都先保留 raw_json，不要猜测后丢数据。
3. 所有外部接口调用都必须带 request_id 落日志。
4. 工作台总表必须由子账户聚合得到，不能直接把工作台账号当成报表主体。
5. 所有代码、文档、配置都必须可维护、可读、可扩展。
```

## 4. 使用原则

以后你只要坚持两句话，OpenClaw 跑偏的概率会低很多：

- `不要重写系统，只在现有官方 open_api 架构上增量扩展。`
- `先给最小变更方案，再实施。`

如果你担心它乱改，再固定补一句：

```text
除非我明确要求，否则不要新增第二套架构，不要改用浏览器抓取，不要破坏现有表结构兼容性。
```
