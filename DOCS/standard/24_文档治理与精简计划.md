# 24 文档治理与精简计划

版本：`v1.0`
状态：`生效中`
最后更新：`2026-03-30`
适用范围：部署、运维、技术标准与运行文档治理
读者对象：后端、运维、联调同事、文档维护者

## 1. 文档定位

本文档用于收口当前 `DOCS/` 中与部署、运维、技术标准、运行事实相关的重复信息、漂移口径和易踩坑点。

它的职责不是替代上位标准，而是给出一份可执行的文档整改清单，统一：

- 哪些文档继续作为真源
- 哪些文档只保留摘要或历史记录
- 哪些主题需要减少重复维护
- 哪些示例内容必须和真实运行状态保持一致

本文档受以下上位文档约束：

- [13_已确认决策基线.md](./13_已确认决策基线.md)
- [03_目标技术架构与部署基线.md](./03_目标技术架构与部署基线.md)
- [17_运维上线清单.md](./17_运维上线清单.md)
- [../README.md](../README.md)

## 2. 当前扫描结论

本轮扫描后的正向结论：

- 正式文档根目录已经统一到 `DOCS/`
- Markdown 相对链接坏链检查结果为 `0`
- 正式文档内已基本清掉本机绝对路径
- 文档优先级在 `DOCS/README.md` 中已经明确

当前仍然需要治理的问题：

1. 同一主题在多份文档里重复维护，开始出现口径漂移
2. 部分长期规范文档仍保留已经过时的接口、发布或配置示例
3. 运行文档里的 `verified_commit` 和真实仓库 `HEAD` 会漂移
4. 文档头模板没有统一，阅读成本偏高
5. 历史文档与内部文档的边界还不够清晰

## 3. 治理原则

1. 同类信息只能保留一个真源，其他文档只做摘要、跳转或历史说明。
2. 标准文档只写稳定规则，不重复维护“当前运行事实”。
3. 运行文档只写“当前真实可用内容”，不回写历史设计目标。
4. 示例必须以代码和真实部署文件为准，不允许凭记忆维护样例。
5. 历史方案、归档计划、代理内部约束必须显式标记，不能与正式标准混放成同等级阅读入口。

## 4. 建议改动与精简清单

### 4.1 接口真源收口

问题：

- [05_内部接口与集成规范.md](./05_内部接口与集成规范.md) 仍维护一套已经过时的路径样例
- 当前实际联调真源已经是 [../runtime/API_REFERENCE.md](../runtime/API_REFERENCE.md)

需要改动：

- `05` 只保留长期接口设计原则、响应规范、边界约束
- 移除当前具体接口路径列表中的过时项
- 所有“当前接口清单”统一指向 `runtime/API_REFERENCE.md`

涉及文件：

- [05_内部接口与集成规范.md](./05_内部接口与集成规范.md)
- [../runtime/API_REFERENCE.md](../runtime/API_REFERENCE.md)

验收标准：

- `05` 中不再出现与当前实现不一致的 `auth/*`、`rankings/*`、`system/health` 等旧路径
- 同事联调时只看 `API_REFERENCE.md` 获取当前接口

示例内容：

```md
本文档只保留内部 API 的长期设计原则、命名规则和边界约束。
当前线上实际可调用接口，请统一参考 `DOCS/runtime/API_REFERENCE.md`。
```

- `09` 只保留确实尚未闭环的外部依赖，删除已拍板内容

涉及文件：

发布、备份、值班执行以 `17_运维上线清单.md` 为真源。
本文仅保留测试验收项，不重复维护持久化目录、备份策略和发布命令。

### 4.3 配置优先级表达拆分

问题：

- 现在多份文档把“代码读取优先级”和“部署真源”混写
- 容易让同事误以为 `config.json` 仍是标准部署入口

需要改动：

- 所有文档统一拆成两个概念：
  - 代码读取优先级
  - 标准部署真源

涉及文件：

- [12_初始化配置模板.md](./12_初始化配置模板.md)
- [../runtime/README.md](../runtime/README.md)
- [../runtime/capabilities/QIANCHUAN_OFFICIAL_API_STANDARD.md](../runtime/capabilities/QIANCHUAN_OFFICIAL_API_STANDARD.md)
- [../runtime/AGENTS.md](../runtime/AGENTS.md)

验收标准：

- 文档里不再出现“`config.json` 是标准部署入口”的误读空间

示例内容：

```md
代码读取优先级：
1. 默认值
2. `config.json`
3. 环境变量

标准部署真源：
1. `.env.dashboard`
2. `docker-compose.dashboard.yml`

```

### 4.4 健康检查说明收成一份真源

问题：

- `healthz / readyz` 在多份文档重复出现
- 示例结构和真实运行返回值容易漂移

需要改动：

- `17` 只保留值班检查步骤
- `API_REFERENCE` 只保留对外接口含义
- `12` 和 `runtime/README` 不再重复维护详细结构示例
- 详细结构以真实 `readyz` 返回为准，示例中避免写死过期版本号

涉及文件：

- [12_初始化配置模板.md](./12_初始化配置模板.md)
- [17_运维上线清单.md](./17_运维上线清单.md)
- [../runtime/README.md](../runtime/README.md)
- [../runtime/API_REFERENCE.md](../runtime/API_REFERENCE.md)

验收标准：

- 任何人只要查一个主入口，就能知道：
  - `/healthz` 只看存活
  - `/readyz` 负责 DB / Redis / Celery / schema 检查

示例内容：

```md
- `GET /healthz`：进程存活检查，只返回 `{"status":"ok"}`
- `GET /readyz`：数据库、Redis、Celery、schema 版本就绪检查
- `readyz` 的字段样例以当前运行返回为准，不在标准文档中写死 schema 数字
```

### 4.5 Runtime 文档校验版本刷新

问题：

- Runtime 文档中的 `verified_commit` 会随着仓库推进而过时
- 文档容易继续描述旧导航或旧接口

需要改动：

- 把 `verified_commit / verified_date` 的维护动作写进文档更新规则
- 每次较大页面、接口、部署方式调整后，同步刷新 runtime 文档

涉及文件：

- [../runtime/README.md](../runtime/README.md)
- [../runtime/STANDARD.md](../runtime/STANDARD.md)
- [../runtime/API_REFERENCE.md](../runtime/API_REFERENCE.md)
- [../runtime/FUNCTIONAL_SPEC.md](../runtime/FUNCTIONAL_SPEC.md)

验收标准：

- Runtime 目录中的核验提交号与当前发布基线一致

示例内容：

```md
verified_commit：`<当前发布基线 commit>`
verified_date：`<本次核验日期>`
核验范围：页面导航、运行接口、健康检查、部署方式
```

### 4.6 内部文档与正式文档边界隔离

需要改动：

- 对内部文档增加 `Internal Only` 或等价说明
- 在总导航里降低其阅读优先级，不放入建议阅读顺序主链

涉及文件：

- [../runtime/AGENTS.md](../runtime/AGENTS.md)

- 联调、运维、产品不会把代理内部约束当作正式开发标准

示例内容：

```md
状态：`Internal Only`
用途：AI / 自动化代理执行约束
不作为研发、运维、联调真源
```

### 4.7 文档头模板统一

问题：

- 许多 `standard/`、`runtime/` 文档头部字段不完整
- 阅读时很难快速判断有效性和适用范围

需要改动：

- 对正式标准和运行文档统一头模板
- 最低要求：
  - `版本`
  - `状态`
  - `最后更新`
  - `适用范围`
  - `读者对象`

涉及文件：

- `DOCS/standard/*.md`
- `DOCS/runtime/*.md`
- `DOCS/runtime/capabilities/*.md`

验收标准：

- 打开任一正式文档，前 10 行内能看到文档用途和当前状态

示例内容：

```md
版本：`v2.1`
状态：`生效中`
最后更新：`2026-03-30`
适用范围：发布、备份、值班、线上检查
读者对象：运维、后端、值班同事
```

### 4.9 补充“标准发布”和“应急热修”边界

问题：

- 当前文档只写了标准 Compose 发布
- 没有明确前端热替换或容器内临时修复的边界

需要改动：

- 在运维真源中明确区分：
  - 标准发布
  - 应急热修
  - 热修后的补偿动作

涉及文件：

- [17_运维上线清单.md](./17_运维上线清单.md)
- [../runtime/README.md](../runtime/README.md)

验收标准：

- 线上紧急修复后，不会遗漏“回写 GitHub / 回写运行目录 / 补一次正式重建”

示例内容：

```md
标准发布：同步文件到线上目录后执行 Compose 重建。
应急热修：仅限紧急止血，可临时替换运行容器内文件。
热修补偿：必须同步回 GitHub、线上目录，并在窗口期补一次标准重建。
```

### 4.10 增加文档自检脚本

问题：

- 当前坏链和绝对路径主要靠人工巡检
- 长期无法保证新增文档继续合规

需要改动：

- 新增一个轻量文档检查脚本
- 固定检查：
  - 相对链接是否存在
  - 是否写入本机绝对路径
  - 正式文档头字段是否缺失
  - `Archived` 文档是否声明替代真源

涉及文件：

- `scripts/check_docs.py` 或同等位置
- [17_运维上线清单.md](./17_运维上线清单.md) 或 [../README.md](../README.md) 中补使用说明

验收标准：

- 任意一次文档更新都能被快速自动检查

示例内容：

```bash
python3 scripts/check_docs.py
```

## 5. 推进顺序

### P0

先做以下 4 项，优先解决误导和漂移：

1. 接口真源收口
2. 发布 / 验收 / 待确认文档去重
3. 配置优先级表达拆分
4. Runtime 校验版本刷新

### P1

再做以下 4 项，解决阅读和执行边界：

1. 健康检查说明收口
2. 内部文档与正式文档隔离
3. 文档头模板统一
4. 历史方案归档写法统一

### P2

最后做制度化收尾：

1. 标准发布与应急热修边界补齐
2. 文档自检脚本落地

## 6. 完成定义

本计划完成时，应满足：

- `DOCS/` 内没有互相冲突的发布、备份、配置真源说法
- 标准接口文档不再维护一套过时 API 路径
- Runtime 文档能明确对应当前发布基线
- 历史文档和内部文档不会再误导同事
- 文档目录新增内容时，有自动检查手段兜底
