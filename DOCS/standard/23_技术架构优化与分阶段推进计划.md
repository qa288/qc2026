## 1. 当前架构事实

### 1.1 运行形态

当前正式运行基线已经稳定为：

- `web`：FastAPI 页面与内部 API
- `worker`：Celery Worker
- `scheduler`：Celery Beat
- `postgres`：主数据库
- `redis`：队列、缓存、锁

当前不做微服务拆分，不引入第二套 Web 技术栈，也不重写成前后端完全分离架构。

### 1.2 当前代码结构事实

截至 `2026-03-25`，运行时代码存在以下明显结构信号：

- `dashboard/main.py`
  - `7107` 行
  - 同时承载：
    - FastAPI app 装配
    - 登录与权限
    - 启动初始化
    - 数据查询
    - 数据同步
    - 上传任务
    - 告警逻辑
    - token 存储
    - 数据库建表与兼容修补
- `dashboard/static/app.js`
  - `3606` 行
  - 同时承载状态管理、API 调用、DOM 渲染和交互逻辑
- `dashboard/main.py`
  - 当前包含 `51` 个路由入口
  - 启动建表脚本覆盖 `28` 类表/索引定义
- `dashboard/db_backend.py`
  - 仍兼容 SQLite 与 PostgreSQL 双后端
- `report_qianchuan.py`
  - 同时承担：
    - 配置读取
    - token 刷新
    - OceanEngine API 接入
    - 上传素材请求封装

### 1.3 当前主要风险

1. 单文件承载过多职责，后续改动冲突概率高。
2. 运行时建表与补列逻辑混在业务代码中，缺少正式迁移体系。
3. 多进程架构下仍存在进程内去重/锁思路，幂等边界不够清晰。
4. 配置读取分散在多个模块，敏感参数和运行参数没有统一口径。
5. 前端未模块化，继续叠功能会快速失控。
6. 现有健康检查、监控、发布流程仍偏“能跑”，不够“可运维”。

## 2. 优化目标

本轮优化只做“在现架构上增量治理”，目标不是推翻重写，而是把当前系统推进到“可持续迭代”的状态。

### 2.1 必须达成的目标

1. 保持 `FastAPI + PostgreSQL + Redis + Celery + Docker Compose` 不变。
2. 页面查询继续只读本地库，不在请求链路直接打官方接口。
3. 生产数据库正式收敛为 PostgreSQL。
4. 新增正式 schema migration 机制。
5. 后端按模块边界拆分，不再继续扩张 `main.py`。
6. 任务调度、上传、历史回补具备明确幂等和可追踪状态。
7. 前端完成最小必要模块化。
8. 安全、监控、发布与回滚流程补齐。

### 2.2 本轮不做的事

- 不拆微服务
- 不重写为 React/Vue SPA
- 不引入 Kafka、ES、分布式链路追踪平台等重基础设施
- 不改变当前产品范围和角色口径

---

## 3. 总体推进原则

1. 先固化底座，再拆业务层。
2. 先降低耦合，再推进大块迁移。
3. 每一阶段必须可运行、可验证、可回滚。
4. 尽量保持接口形态与页面行为不变。
5. 所有结构性改动都必须同步更新标准文档。

## 4. 工作流分组

### 4.1 工作流 A：配置与运行参数治理

目标：

- 配置读取集中化
- 环境变量与默认值统一
- 为后续 secrets 管理和多环境配置打底

任务：

- 新增统一配置模块，收口：
  - 路径
  - 数据库
  - Redis/Celery
  - 登录引导账号
  - token 缓存
  - 调度频率
  - session 配置
- 将 `web` 与 `scheduler` 共享的配置从各自文件内联读取迁出
- 明确：
  - 开发默认值
  - 生产必须值
  - 敏感字段

完成标准：

- 不再在多个模块中各自解析同一组环境变量
- 配置变更可通过单点模块审计

### 4.2 工作流 B：数据库 schema 与 migration

目标：

- 把“业务逻辑 + 建表脚本”分离
- 从运行时建表，过渡到正式 migration

任务：

- 抽离基础 schema 真源
- 引入 migration 目录与版本号
- 把新增表、索引、补列逻辑迁移到正式 migration
- 将启动流程从“直接建表”收敛为“校验版本并应用迁移”

完成标准：

- schema 变更不再写进 `DashboardService`
- 线上 schema 变更可审计、可回滚

### 4.3 工作流 C：后端模块化拆分

目标：

- 将当前重单体拆成清晰层次

目标目录：

- `dashboard/app.py`
- `dashboard/routers/`
- `dashboard/services/`
- `dashboard/repositories/`
- `dashboard/policies/`
- `dashboard/schemas/`
- `dashboard/jobs/`
- `dashboard/integrations/`

任务：

- 先抽 `settings / schema / auth / repositories`
- 再拆：
  - session 与用户管理
  - performance 查询
  - material 查询
  - alert 规则
  - upload 任务
  - token 管理

完成标准：

- `main.py` 只保留：
  - app 初始化
  - middleware
  - router 注册
  - startup/shutdown

### 4.4 工作流 D：任务调度与幂等治理

目标：

- 让 `sync / detail_sync / backfill / upload / alerts` 在多进程下具备稳定幂等

任务：

- 用 Redis 或数据库实现：
  - 分布式锁
  - 幂等键
  - 去重窗口
- 明确每类任务的：
  - 入队参数
  - 执行状态
  - 失败重试
  - 人工重跑方式

完成标准：

- 不再依赖进程内状态作为唯一去重手段
- 同一类任务重复触发时结果可预测

### 4.5 工作流 E：OceanEngine 接入层治理

目标：

- 把外部 API 接入与业务层分离

任务：

- 抽出独立 `integrations/oceanengine`
- 统一：
  - token 读取与刷新
  - 错误映射
  - 超时/重试
  - 上传接口包装
  - 权限不足错误文案

完成标准：

- 业务层不直接感知 HTTP 细节
- 官方接口相关改动可在独立模块内完成

### 4.6 工作流 F：前端模块化

目标：

- 在不切换技术栈的前提下，让前端可持续维护

任务：

- `app.js` 拆为：
  - `api/`
  - `state/`
  - `views/overview`
  - `views/accounts`
  - `views/plans`
  - `views/materials`
  - `views/uploads`
  - `views/access`
  - `views/signals`
  - `utils/`
- 将公共表格、筛选器、时间范围处理收口复用

完成标准：

- 单视图改动不再牵动整份前端主脚本

### 4.7 工作流 G：权限与安全

目标：

- 把“角色能看什么、能做什么”做成统一规则层

任务：

- 统一管理员、主管、运营 policy
- 登录限流
- 安全 Cookie
- CSRF 防护
- 登录审计
- 敏感接口权限测试

完成标准：

- 权限判断不再散在每个接口内手写
- 登录与会话达到生产最低安全基线

### 4.8 工作流 H：可观测性与运维

目标：

- 系统不只“能运行”，还要“可判断是否健康”

任务：

- 拆分：
  - `healthz`
  - `readyz`
  - DB/Redis/Celery 检查
- 增加结构化日志
- 增加关键指标：
  - 主同步成功率
  - 细粒度同步耗时
  - backfill 排队量
  - 上传成功率
  - alert 派发结果

完成标准：

- 出问题时可快速判断是应用、数据库、Redis、任务队列还是外部接口问题

### 4.9 工作流 I：测试与交付

目标：

- 建立可安全重构的最小自动化保障

任务：

- 增加：
  - repository tests
  - service tests
  - API smoke tests
  - Celery task smoke tests
  - OceanEngine mock tests
- 建立发布流程：
  - build
  - migrate
  - deploy
  - verify
  - rollback

完成标准：

- 关键结构性调整前后都有回归保障
- 发布不再依赖手工口头流程

## 5. 分阶段执行计划

### 阶段 0：基线冻结与摸底

目标：

- 冻结当前运行口径
- 明确重构边界

任务：

- 统计当前路由、表、任务、同步频率、角色能力
- 明确不可变约束
- 整理本计划文档

退出标准：

- 现状与目标一致收口

### 阶段 1：底座收口

目标：

- 先拆最容易引发后续冲突的基础层

任务：

- 配置读取集中化
- 基础 schema 抽离
- migration 骨架建立
- 健康检查增强

退出标准：

- 配置、schema、启动初始化不再继续膨胀在 `main.py`

### 阶段 2：后端结构拆分

目标：

- 明确 Web 层、Service 层、Repository 层边界

任务：

- 拆 `auth / users / permissions`
- 拆 `dashboard / performance / materials / alerts`
- 拆 `token / upload / sync`

退出标准：

- `main.py` 降到装配层角色

### 阶段 3：任务链路加固

目标：

- 让同步、回补、上传、告警在多进程下稳定运行

任务：

- 锁、幂等、重试、状态收口
- 清理进程内去重逻辑

退出标准：

- 任务重复触发不会产生不可控副作用

### 阶段 4：前端模块化

目标：

- 降低前端改动面和联动风险

任务：

- 拆 API 层、状态层、视图层
- 收口公共渲染逻辑

退出标准：

- 前端改动按模块分治

### 阶段 5：安全、监控、测试、发布

目标：

- 把系统带到“可长期维护”的基线

任务：

- 安全基线
- readyz/metrics/logging
- 自动化测试
- 发布/回滚流程

退出标准：

- 能稳定迭代并可控上线

## 6. 依赖顺序

必须按以下顺序推进：

1. `配置治理`
2. `schema/migration`
3. `后端拆分`
4. `任务幂等与锁`
5. `前端拆分`
6. `安全/监控/测试/发布`

原因：

- 如果没有统一配置与 schema 真源，后续拆模块会不断重复搬运脏逻辑。
- 如果没有 migration，后续拆 repository 仍会把数据结构改动耦合进业务层。
- 如果没有任务幂等，后续做上传和回补优化时风险会持续扩大。

---

## 7. 每阶段交付要求

每个阶段都必须产出：

- 代码改动
- 文档回写
- 验证记录
- 下一阶段入口条件

禁止：

- 只改代码不改文档
- 只写计划不落首个执行切片
- 一次性跨越多个阶段并混合高风险改动
