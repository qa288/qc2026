# 03 目标技术架构与部署基线

适用范围：技术栈、部署基线、数据与运行约束
读者对象：后端、运维、架构治理参与者

## 1. 目标

本文档定义 V1 的技术与部署标准。
本文档受 [13_已确认决策基线.md](./13_已确认决策基线.md) 约束，并高于运行事实文档与计划文档。

---

## 2. 统一技术栈

第一阶段正式技术栈固定为：

- Web/API：`FastAPI`
- 主数据库：`PostgreSQL`
- 缓存 / 队列 / 分布式锁：`Redis`
- 异步任务：`Celery`
- 定时调度：独立 `scheduler` 进程，可采用 `Celery Beat`
- 部署：`Docker Compose`
- 前端：保留当前轻前端，不强制前后端分离
- 反向代理与运维入口：`1Panel`

部署职责边界：

- `1Panel`
  - 反向代理
  - 域名与证书
  - 站点入口与容器观察
- `Docker Compose`
  - 应用服务编排真源
  - `web / worker / scheduler / postgres / redis` 的正式启动方式

禁止只在 `1Panel` 页面手工改容器参数而不回写 Compose 文件。

以下方案降为历史方案，不再作为正式要求：

- `Next.js`
- `NestJS`
- `Prisma`
- `BullMQ`
- `TypeScript Monorepo`

## 3. 服务边界

运行角色统一如下：

### 3.1 web

职责：

- 登录与会话
- 页面渲染
- 内部 API
- 只读查询

禁止：

- 在 Web 进程里承担正式业务调度

### 3.2 worker

职责：

- 官方接口采集
- 数据落库
- 聚合计算
- 告警评估与通知任务执行

### 3.3 scheduler

职责：

- 发起定时任务
- 不承担页面请求

### 3.4 postgres

职责：

- 结构化数据真源

### 3.5 redis

职责：

- Celery Broker
- 缓存
- 分布式锁
- Token 刷新锁

## 4. 数据与存储分层

### 4.1 PostgreSQL

保存：

- 账户、计划、商品、素材结构化数据
- 快照与聚合结果
- 用户、规则、告警、通知、任务日志
- Token 元数据

查询规则：

- 页面查询默认只读 PostgreSQL
- 不允许在用户请求生命周期内直接拉取远程报表作为页面返回
- 如果发现历史日快照缺失，只允许异步排队后台回补
- 页面必须先返回当前库内已有结果，不得同步等待远程接口完成

### 4.2 本地卷 / 对象存储

保存：

- 原始接口回包
- 导出文件
- 调试快照

第一阶段允许先使用本地卷；后续可切对象存储。

### 4.3 Token 缓存文件

规则：

- PostgreSQL 为主真源
- 本地文件为缓存副本
- 推荐路径：`/app/data/qianchuan_latest_token.json`

## 5. Token 标准

### 5.1 刷新规则

- 只允许一个实例刷新 token
- 刷新成功后必须立即保存新的 `refresh_token`
- 其它实例只能读取最新 token

### 5.2 共享方式

标准方式：

- `GET /api/system/integrations/ocean-engine/token-latest`

补充方式：

- 只读缓存文件

禁止：

- 多台服务器各自刷新同一套 `refresh_token`

## 6. 账户汇总降级标准

标准链路：

1. 优先调用官方账户汇总接口
2. 若返回 `50000` 或持续不可用
3. 自动降级为计划聚合账户汇总

要求：

- 明确标记来源
- 不得把降级值伪装成官方原始值

## 6.1 历史数据回补标准

标准链路：

1. 页面或内部查询请求命中 `昨日 / 近7天 / 近30天 / 指定日期范围`
2. 服务端先检查本地是否存在对应日级快照
3. 若缺失，则异步提交按天回补任务
4. 页面先返回当前库内结果，并标记“历史补齐中”
5. 回补写入 PostgreSQL 后，后续查询继续只读库

要求：

- 回补是“补本地库”，不是“直接把远程响应返回给页面”
- 回补后要清理相关查询缓存
- 同一日期的回补必须幂等

## 6.2 重查询榜单预聚合标准

以下结果优先采用预聚合，而不是每次临时扫明细表：

- 素材榜
- 商品榜
- 员工榜（后续可扩）

要求：

- 同步任务写原始明细后，再写预聚合结果
- 页面查询优先读取预聚合表
- 若老数据还未生成预聚合结果，可临时回退到原始明细聚合，但应尽快回填

## 7. 通知基线

第一阶段只保留：

- 阈值告警

通知通道：

- 系统 -> 通知适配层 -> OpenClaw / 其它渠道

后续扩展：

- Feishu
- DingTalk
- Email / Slack / Telegram

第一阶段不做：

- 定时简报
- 日报 / 周报
- 多通道联动

## 7.1 手工同步标准

手工同步接口必须：

- 只负责提交任务到队列
- 立即返回 `202 Accepted`
- 由 Worker 异步执行

禁止：

- Web 请求阻塞等待同步完成
- 让反向代理承担长任务等待

---

## 8. 资源基线

### 8.1 试运行 / 联调

- `2C4G`

### 8.2 正式生产（目标 1000 账户）

建议至少：

- `8C16G`
- SSD 持久化卷
- PostgreSQL、Redis、应用容器独立资源控制

说明：

- `2C4G` 只适合作为试运行起步配置
- 不应作为 `1000` 账户正式容量基线

## 8A. 线上目录约束

当前线上部署目录不是 Git 工作树。

要求：

- 代码真源在 GitHub 仓库
- 线上发布通过“同步文件 + Compose 重建”完成
- 不依赖线上目录直接 `git pull`

- 站点反向代理
- HTTPS 证书管理
- Docker / Compose 应用管理

- 文档中提到的 `Docker Compose` 仍是应用编排基线
